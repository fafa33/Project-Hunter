## Blocking finding

### P1 — name-mangled write helpers remain directly callable

`SupplyAndValueCaptureService` now owns `__insert_receipt`, `__insert_record`, `__table_for`, and `__persist_authorized`, but Python double-underscore name mangling is not an authority boundary. Any in-process caller can invoke these attributes through names such as:

- `SupplyAndValueCaptureService._SupplyAndValueCaptureService__insert_receipt`
- `SupplyAndValueCaptureService._SupplyAndValueCaptureService__insert_record`
- `SupplyAndValueCaptureService._SupplyAndValueCaptureService__table_for`
- `service._SupplyAndValueCaptureService__persist_authorized`

The two static insert helpers accept a caller-supplied SQLite connection and record/receipt. A caller can therefore open the repository database and call the mangled insert helpers directly, bypassing provider verification, source-registry authorization, endpoint/parser/capability checks, evidence authorization, correction transition policy, receipt signature/hash verification, and receipt-record binding.

The boundary test only rejects the former module-level helper names. It does not check the mangled class attributes and therefore does not prove that direct write capability is unavailable.

## Required remediation

- Remove reusable write helpers from importable/class-addressable Python objects.
- Put authoritative persistence behind an actual process boundary, database permission boundary, or a dedicated persistence component whose credentials/capability are not available to ordinary callers.
- At minimum, tests must demonstrate that ordinary application code cannot obtain or invoke any raw receipt/record insertion primitive, including mangled attributes.
- Repeat the independent audit on the resulting `main` commit.

v3.5.0 must not be closed while this finding remains unresolved.
