# v3.5.0 Authority Receipt Remediation

## Scope

This change closes the four authority blockers identified by the final v3.5.0 remediation audit. It does not introduce valuation, comparative valuation, mispricing, asymmetry, Opportunity scoring, Portfolio Intelligence, Dashboard calculations, or Market Validation inputs.

## Controls

- Replaced the module-level provider seal with per-provider cryptographic signing keys.
- Provider results now contain an immutable acquisition receipt and signature.
- Service ingestion requires the originating registered provider and verifies the signature, payload hash, receipt hash, registry fingerprint, endpoint, parser, capability, source, and acquisition kind.
- Added SQL-authoritative `value_capture_acquisition_receipts` persistence.
- Acquisition receipts and derived records are inserted atomically.
- Records are bound to receipts by acquisition ID, raw payload hash, source, parser, identity, and acquisition chronology.
- Removed `_SERVICE_WRITE_TOKEN`, `_service_commit`, and repository `_connect()` from the supported architecture.
- The public repository object is read-only; the service receives a closure-scoped authority writer during composition.
- Added explicit registry correction-transition policy.
- Cross-source corrections require an allowed predecessor tier and cannot downgrade authority.

## Tests

Deterministic tests cover:

- payload tampering;
- cross-provider signature forgery;
- read-only repository API;
- atomic receipt and record persistence;
- receipt-to-record provenance binding;
- future-known evidence rejection;
- temporal chronology;
- branching correction rejection;
- strict-known replay before and after correction;
- correction authority downgrade rejection.

## Verification

Completion requires Ruff, Black, mypy, pytest, diff checks, and a separate post-implementation architecture audit.
