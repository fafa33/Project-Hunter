## Blocking finding

### B1 — Authoritative write capability remains caller-constructible and directly reachable

`src/hunter/value_capture/authority.py` defines the importable class `_ValueCaptureAuthorityWriter`. Its constructor accepts any `SupplyAndValueCaptureRepository` and any caller-created `ValueCaptureVerificationKeyRegistry`. Its public `persist()` method then writes authoritative receipts and records directly through repository module internals.

The writer validates only that the receipt is signed by a key contained in the caller-supplied verification registry and that the receipt binds to the record. It does not validate the receipt against the canonical `ValueCaptureSourceRegistry`, provider/source registration, endpoint authorization, parser authorization, capability authorization, correction-transition authority, or evidence requirements enforced by `SupplyAndValueCaptureService`.

Therefore an arbitrary in-process caller can:

1. create its own HMAC key;
2. create a matching verification-key registry;
3. construct and sign a receipt and normalized record;
4. instantiate `_ValueCaptureAuthorityWriter` directly; and
5. persist authoritative state without passing through the service authority checks.

The leading underscore and omission from package `__all__` are naming conventions, not an enforcement boundary in Python. This preserves the same semantic bypass that the remediation intended to remove, only moved from the repository object into another importable module.

Required closure:

- authoritative persistence capability must not be constructible from caller-supplied repository and keys;
- writer creation must be sealed behind a module-owned capability/factory unavailable to normal callers, or persistence must require an unforgeable service-issued authorization object whose construction is not exported or caller-controlled;
- the persistence boundary must independently enforce or cryptographically bind canonical source-registry authority, not merely trust a caller-created key registry;
- deterministic tests must demonstrate that direct imports/instantiation and caller-created keys cannot bypass service registry, evidence, correction, and capability checks.

## Result

**BLOCKED**

v3.5.0 cannot be declared closed because the authoritative write boundary remains bypassable by ordinary in-process code. No valuation, scoring, Opportunity, Dashboard, or Market Validation functionality was evaluated or activated by this audit.
