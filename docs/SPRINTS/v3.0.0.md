# Sprint v3.0.0: Evidence Intelligence Foundation

## Status

- Sprint: Evidence Intelligence Foundation
- Release: `v3.0.0`
- Status: approved for implementation planning
- Architecture source: approved Evidence Intelligence Layer architecture from the v3.0.0 architecture review thread

## Mission

Implement the first production Evidence Intelligence Layer so Hunter can transform immutable evidence into structured, reusable, evidence-backed knowledge without changing the canonical runtime.

The sprint establishes the foundations for future Competition Intelligence, Developer Intelligence, Network Intelligence, Intrinsic Value, and Opportunity Engine work.

## Architectural Invariants

- Raw evidence remains immutable.
- AI is a processor, never a source of truth.
- Candidate Registry remains canonical for investable identity.
- Identity and Trust Layer remains the authority boundary for verified identity.
- SQL is authoritative storage for structured knowledge.
- Knowledge is claim-centered.
- `KnowledgeClaim` is the only canonical structured truth record.
- `KnowledgeRelationship` is only a graph-ready projection of eligible claims.
- Document lifecycle, source authority, claim lifecycle, and conflict lifecycle are append-only and time-aware.
- Strict historical replay uses both `effective_at <= cutoff` and `recorded_at <= cutoff`.
- Reconstruction mode must never be reported as knowledge known at cutoff.
- No scoring, ranking, valuation, committee, Opportunity Timing, Market Validation, or `EvidenceBackedProjectExecutor` behavior changes.

## Implementation Phases

### Phase 1: Models and Predicate Registry

Deliverables:

- `EvidenceDocument`
- `EvidenceSpan`
- `DocumentLifecycleEvent`
- `SourceAuthorityVerificationEvent`
- `KnowledgeEntity`
- `PredicateRegistry`
- `KnowledgeClaim`
- `ClaimLifecycleEvent`
- `KnowledgeConflict`
- `ConflictLifecycleEvent`
- `KnowledgeRelationship` projection model

Acceptance criteria:

- All models validate required fields.
- Claim statuses exclude `resolved`.
- Conflict statuses include `resolved`.
- Relationship projection requires `claim_id`.
- Predicate definitions are versioned.
- No model uses undefined multi-valued SQL-authoritative fields for lineage.

### Phase 2: SQL Repository and Migrations

Deliverables:

- SQL tables for documents, spans, authority events, entities, predicates, claims, lifecycle events, conflicts, relationship projections, provider artifacts, processing runs, checkpoints, and security audit events.
- Normalized association tables for evidence, span, claim, conflict, entity, predicate, and lifecycle-event links.
- Required indexes for point-in-time, lineage, predicate, lifecycle, and reporting queries.

Acceptance criteria:

- All writes are idempotent.
- Current-state fields are materialized only from append-only events.
- Complete claim lineage can be reconstructed without scanning JSON/list fields.
- Existing repositories and runtime behavior remain backward compatible.

### Phase 3: Evidence Intake, Lifecycle, and Authority

Deliverables:

- Evidence document intake from existing persisted evidence references.
- Stable normalized rendition metadata.
- Evidence span creation.
- Document lifecycle event append.
- Source authority verification event append.

Acceptance criteria:

- Raw evidence is never modified.
- Document lifecycle is reconstructable at historical cutoff.
- Authority is reconstructable at historical cutoff.
- Strict known-by-Hunter authority requires `recorded_at <= cutoff`.
- Current authority does not leak into historical confidence.

### Phase 4: Provider Interface and Security Boundary

Deliverables:

- AI provider abstraction.
- Provider health and unavailable states.
- Versioned extraction schemas.
- Prompt-injection detection.
- Security audit events.

Acceptance criteria:

- Provider output is treated only as extraction proposal.
- Providers cannot invoke tools, fetch sources, write repositories, alter schemas, or change configuration.
- Provider failure creates explicit unavailable state.
- No provider-specific logic leaks outside adapters.

### Phase 5: Classification and Extraction Validation

Deliverables:

- Deterministic evidence classification.
- Entity extraction validation.
- Claim extraction validation.
- Literal support validation.
- Semantic support validation.
- Predicate Registry validation.

Acceptance criteria:

- Unsupported predicates are rejected.
- Co-mention does not create claims or relationships.
- Numerical values, dates, addresses, identifiers, URLs, and direct quotes require literal support.
- Semantic support is accepted only when evidence explicitly supports the claim.
- Unsupported, inferred, predictive, causal, ownership, or investment conclusions are rejected.

### Phase 6: Claim Persistence, Lifecycle, and Confidence

Deliverables:

- Canonical claim persistence.
- Claim evidence and span associations.
- Claim lifecycle event append.
- Deterministic confidence projection.
- Point-in-time claim status queries.

Acceptance criteria:

- Claim persistence and lifecycle event append are atomic.
- Historical claim status is reconstructed from `ClaimLifecycleEvent`.
- Strict known-by-Hunter replay uses `effective_at` and `recorded_at`.
- Current claim status, confidence, `superseded_at`, and `retracted_at` are denormalized projections.

### Phase 7: Conflict Detection and Lifecycle

Deliverables:

- Predicate-aware conflict detection.
- Conflict persistence.
- Conflict evidence, span, and claim associations.
- Conflict lifecycle event append.
- Point-in-time conflict status queries.

Acceptance criteria:

- Different values are not automatically conflicts.
- Conflict detection respects subject, predicate, scope, qualifiers, validity period, polarity, modality, authority, document lifecycle, correction, and retraction state.
- Conflict resolution belongs only to `KnowledgeConflict`.
- Claim status changes caused by conflict resolution use valid claim lifecycle events.

### Phase 8: Relationship Projection

Deliverables:

- Relationship projection from eligible entity-to-entity claims.
- Projection refresh/rebuild path.
- Point-in-time relationship view.

Acceptance criteria:

- Relationship projections have no independent truth, lifecycle, confidence, conflict, or versioning.
- Relationship status and confidence derive from canonical claim state.
- Historical relationship views reconstruct through claim, document, authority, and conflict events.

### Phase 9: CLI, Reporting, and Automation

Deliverables:

- Evidence Intelligence CLI namespace.
- Coverage report.
- Source authority report.
- Document lifecycle report.
- Claim lifecycle report.
- Conflict lifecycle report.
- Candidate explain report.
- Security audit report.
- Idempotent automation jobs using the existing Scheduler.

Acceptance criteria:

- Reports distinguish current, historical, strict known-by-Hunter, and reconstructed-after-cutoff states.
- Reconstruction mode is never reported as known at cutoff.
- Scheduler remains operational-only.
- Analytical dependency order is owned by the Evidence Intelligence pipeline.

## Migration and Backward Compatibility Requirements

- Additive migrations only.
- No destructive rewrite of existing evidence.
- No migration of Candidate Registry identity semantics.
- No change to Discovery, Market Validation, scoring, committee, Opportunity Timing, or `EvidenceBackedProjectExecutor`.
- No runtime dependency on an AI provider for existing commands.
- Provider credentials must be optional and environment-driven.
- Historical replay must preserve no-lookahead behavior.
- Runtime data and large provider artifacts must not be committed to Git.

## Completion Gate

Sprint v3.0.0 is complete only when:

- Evidence Intelligence stores structured knowledge as canonical claims.
- Raw evidence remains immutable.
- Document lifecycle is append-only and replay-safe.
- Source authority is append-only and replay-safe.
- Claim lifecycle is append-only and replay-safe.
- Conflict lifecycle is append-only and replay-safe.
- Relationship projections derive only from claims.
- All lineage is SQL-normalized.
- Strict known-by-Hunter replay prevents future-data leakage.
- AI output is validated before persistence.
- Reports expose evidence, authority, lifecycle, conflicts, missing evidence, and reconstruction state.
- All required tests and quality gates pass.
- Existing production behavior remains backward compatible.
