# Sprint v2.7.0: Global Discovery and Candidate Registry

## Status

- Sprint: Global Discovery and Candidate Registry
- Release: `v2.7.0`
- Status: released
- Release tag: `v2.7.0`

## Read First

The sprint followed the Hunter architecture manifest, architecture specification, roadmap, implementation guide, and release-specific implementation prompt.

## Mission

Implement the first production Discovery Layer so Hunter becomes a market discovery engine before it is a project analysis engine.

## Business Objective

Hunter needed to move beyond a manually configured 50-project universe and begin producing a defensible market-wide triage list of candidates that deserve deeper research.

## Scope

- Discovery adapter framework.
- Live CoinGecko market discovery.
- Live DefiLlama protocol discovery.
- SQL-backed Candidate Registry.
- Seed import of the existing 50 configured projects.
- Deterministic candidate identifiers.
- Source identifiers, aliases, and source references.
- Candidate lifecycle states.
- Lightweight screening.
- Persistent candidate queue.
- Discovery automation jobs.
- Discovery CLI commands.
- Coverage and report surfaces.

## Out of Scope

- Replacement of `EvidenceBackedProjectExecutor`.
- Scoring, weighting, committee, timing, replay, historical validation, or calibration changes.
- Intrinsic Value.
- Competition Intelligence.
- Network Effects.
- Portfolio management.
- REST API.
- Dashboard redesign.

## Engineering Principles

- Treat configured projects as a compatibility seed, not the complete market.
- Preserve deterministic identifiers and idempotent registry writes.
- Preserve source provenance for discovered candidates.
- Never merge candidates by ticker equality.
- Keep discovery separate from deep investment scoring.

## Performance

- Use SQL-backed indexed persistence for registry state.
- Avoid JSONL as the authoritative market-wide registry.
- Keep screening cheap enough for thousands of candidates.
- Preserve incremental extension paths for future provider growth.

## Automation

- Add idempotent discovery automation jobs.
- Preserve Scheduler as operational infrastructure.
- Keep analytical execution out of Scheduler.
- Persist run status and expose discovery status through CLI.

## Testing

mypy

## Acceptance Criteria

- Discovery Layer implemented without replacing the canonical runtime.
- Candidate Registry populated from configured seed projects.
- CoinGecko and DefiLlama discovery operational.
- Screening and queue surfaces available.
- Discovery coverage reporting available.
- Quality gates pass.

## Definition of Done

The sprint is complete when Hunter has a durable market-wide discovery entry point, seeded configured projects, live provider discovery, lightweight screening, a persistent queue, documentation, validation, commit, push, and release tag.

## Success Metrics

- Configured universe imported.
- Live provider records observed.
- Unique candidate count exceeds the static project universe.
- Queue entries generated.
- Coverage report distinguishes registry, discovery, screening, analyzability, and deep-analysis readiness.
