# Sprint v2.7.0: Global Discovery and Candidate Registry

## Status

- Release: `v2.7.0`
- Status: released
- Release tag: `v2.7.0`

## Mission

Implement the first production Discovery Layer so Hunter becomes a market discovery engine before it is a project analysis engine.

The configured 50-project universe becomes a compatibility seed, not the complete investable universe.

## Implemented Scope

- Discovery adapter framework.
- Live CoinGecko market discovery.
- Live DefiLlama protocol discovery.
- SQL-backed Candidate Registry.
- Seed import of the existing 50 configured projects.
- Deterministic candidate identifiers.
- Source identifiers, aliases, and source references.
- Candidate lifecycle states.
- Lightweight screening.
- Persistent candidate queue.
- Discovery automation jobs.
- Discovery CLI commands.
- Coverage and report surfaces.

## Architecture Boundaries

- Did not replace `EvidenceBackedProjectExecutor`.
- Did not change scoring, weighting, committee logic, timing algorithms, replay, historical validation, or calibration.
- Did not implement intrinsic value, competition intelligence, network effects, portfolio management, REST API, or dashboard redesign.

## Production Behavior

- Registry state is stored in ignored runtime SQLite storage under `data/discovery/runtime/candidates.sqlite`.
- Candidate writes are idempotent.
- Ticker equality is not used for merges.
- Provider failures are represented explicitly.
- Discovered candidates remain pre-analysis candidates until evidence supports advancement.

## Released Capabilities
