## Read First

The sprint followed the Hunter architecture manifest, architecture specification, roadmap, implementation guide, and `v2.7.1` hardening instructions.

## Mission

Harden the Discovery Layer created in `v2.7.0` so it behaves deterministically under live provider data, provider failure, repeated sync, and larger imports.

## Business Objective

Make discovery results trustworthy enough for real market triage by eliminating registry corruption risks, provider failure ambiguity, and weak-evidence advancement.

## Scope

- Deterministic batch registry upserts.
- One SQLite transaction per provider import.
- Canonical seeded identity preservation when provider identifiers merge into seeded candidates.
- Provider retry and backoff.
- Explicit provider failure isolation.
- Deterministic screening outcomes: `advanced`, `deferred`, and `rejected`.
- Queue-score clamping.
- Expanded discovery coverage reporting.
- Additional deterministic registry tests.

## Out of Scope

- Discovery Layer redesign.
- Identity Resolution engine.
- DEX-market discovery adapters.
- Canonical scoring, committee, timing, replay, historical validation, or calibration changes.

## Engineering Principles

- Prefer deterministic idempotency over broad refactoring.
- Preserve canonical seeded candidate identity.
- Do not treat provider failure as absence of market activity.
- Do not advance candidates that lack minimum evidence.
- Keep changes narrowly scoped to discovery hardening.

## Performance

- Replace one-connection-per-candidate import behavior with one transaction per batch.
- Increase registry list limits for larger operational inspection.
- Preserve indexed identifier and slug lookup behavior.
- Add batch iteration support for future scale.

## Automation

- Preserve existing discovery automation jobs.
- Ensure queue refresh remains idempotent.
- Keep provider failure states visible through persisted discovery runs.

- Large candidate imports are idempotent.
- Quality gates pass.
