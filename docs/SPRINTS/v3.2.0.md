## Mission

Implement a cross-cutting Data Sufficiency and Degraded Mode Layer so Hunter can state, for every supported analysis purpose, which required evidence is available, stale, partial, unavailable, direct, proxy, conflicting, or insufficient.

The sprint improves decision quality by preventing missing or weak evidence from being silently interpreted as project weakness, analytical completeness, or investment confidence.

## Business Objective

Hunter should become better at answering:

> Can this candidate or analysis output be trusted enough for the decision being made, and what evidence is missing or degraded?

This sprint does not make Hunter more optimistic or pessimistic. It makes Hunter more honest about evidence limits.

## Architectural Invariants

- Missing data is never fabricated, guessed, silently substituted, or treated as negative project evidence.
- Proxy signals are never represented as missing direct observations.
- Analytical outputs and evidence sufficiency/confidence remain separate.
- Data sufficiency may annotate and gate reports, but it must not change scoring, ranking, valuation, committee decisions, Opportunity Timing, Market Validation, or canonical runtime behavior in this sprint.
- Candidate Registry remains canonical for investable identity.
- Identity/Trust remains authoritative for identity and source trust state.
- Evidence Intelligence claims remain canonical structured truth.
- Competitive Intelligence peer and competitor outputs remain unchanged except for additive sufficiency annotations.
- SQL remains authoritative for structured sufficiency state.
- Strict historical replay uses both effective-time and recorded-time constraints.
- Reconstruction-after-cutoff must be explicitly labeled and never presented as known at cutoff.
- Free/public sources may be optional evidence inputs, but no paid provider or distributed infrastructure is introduced.

### `DegradedModePolicy`

A versioned policy that determines how Hunter reports when required evidence is stale, partial, unavailable, conflicted, or proxy-only.

Required policy outcomes:

- `normal`
- `degraded_non_blocking`
- `degraded_material_limitation`
- `blocked_insufficient_evidence`
- `unavailable`

The policy may suppress or annotate a report field, but it must not substitute values or alter canonical scoring/ranking behavior in this sprint.

## Evidence Semantics

Direct observations include evidence records that measure the target concept directly, such as verified market data for market fields, validated on-chain observations for on-chain capital flow, accepted Evidence Intelligence claims for structured facts, or authoritative registry identity for identity fields.

Proxy signals are indirect indicators that may support context but cannot be persisted or reported as the missing direct observation. Every proxy must retain:

- proxy type;
- source evidence id;
- span id when available;
- policy allowing proxy use;
- limitation text;
- replay mode;
- confidence impact.

Cross-source validation records agreement, disagreement, missing-provider state, freshness differences, source authority, and conflict links. Disagreement must not automatically mean project weakness; it is a data-quality state.

## Data Flow

```text
Registered requirements
  + Candidate Registry identity
  + Identity/Trust state
  + Evidence Intelligence claims, documents, authority, and conflicts
  + Competitive Intelligence outputs
  + existing engine evidence and provider status
  -> Availability evaluation
  -> Cross-source validation and disagreement detection
  -> Sufficiency assessment
  -> Degraded mode policy decision
  -> SQL persistence with normalized lineage
  -> CLI, reports, automation, and advisory committee/report annotations
```

## Class Boundaries

Expected production components:

- `DataRequirementRegistry`: registers and reads versioned requirements without scanning all candidates.
- `DataAvailabilityEvaluator`: evaluates direct, proxy, stale, partial, and unavailable states for selected candidates and requirements.
- `CrossSourceValidationService`: compares compatible source observations and records disagreements.
- `DataSufficiencyAssessor`: aggregates availability into reportable sufficiency assessments.
- `DegradedModePolicyEngine`: applies versioned degraded-mode rules.
- `DataSufficiencyRepository`: SQL-authoritative persistence and point-in-time queries.
- `DataSufficiencyReporter`: reports coverage, missing evidence, limitations, disagreements, and replay state.
- `DataSufficiencyAutomationManager`: installs operational jobs into the existing Scheduler format.

No component may call or modify scoring, ranking, Opportunity Timing, committee, valuation, Market Validation decisions, or canonical runtime behavior.

## Historical Replay

All point-in-time queries must support:

- current mode;
- historical strict known-by-Hunter mode;
- reconstructed-after-cutoff mode.

Strict known-by-Hunter mode may use only requirement definitions, evidence, claims, authority, lifecycle, conflicts, provider status, competitive records, availability records, and assessments with `effective_at <= cutoff` and `recorded_at <= cutoff`.

Reconstruction mode may use later-recorded records whose effective time is within the cutoff, but reports must label the output `reconstructed_after_cutoff` and must not present it as known by Hunter at the cutoff.

Current sufficiency projections are query conveniences only. Historical reports must reconstruct from point-in-time eligible records.

## Automation

Install idempotent jobs using the existing Scheduler configuration model:

- data requirement validation;
- availability refresh;
- stale evidence detection;
- cross-source disagreement detection;
- sufficiency assessment refresh;
- degraded-mode report refresh.

Scheduler remains operational-only. Analytical dependency order belongs to the Data Sufficiency services, not the Scheduler.

Jobs must be idempotent, checkpointed, bounded, non-overlapping, retry-safe, restart-safe, and failure-isolated. Missing provider data must produce explicit unavailable or stale states, never zero or negative evidence.

## Performance

The implementation must scale toward tens of thousands of candidates and many engine requirements.

Requirements:

- select candidates by indexed candidate id, lifecycle state, changed evidence, changed provider status, or checkpoint;
- evaluate incrementally from changed requirements, changed evidence, or stale thresholds;
- avoid full registry scans during normal automation;
- batch writes;
- preserve idempotency keys;
- avoid O(n²) cross-source comparisons except inside bounded candidate/source groups;
- support per-engine and per-purpose filtered queries.

## Implementation Phases

### Phase 1: Models, Policies, and Requirement Registry

Deliverables:

- Data sufficiency domain models.
- Availability, sufficiency, directness, proxy, disagreement, and degraded-mode enums.
- Versioned `DataRequirement` and `DegradedModePolicy` definitions.
- Deterministic id generation.
- Documentation of direct versus proxy semantics.

Acceptance criteria:

- Models validate required fields and enum values.
- Proxy-allowed requirements explicitly define accepted proxy types.
- Required direct observations cannot be satisfied by proxy signals.
- Policy versions are stable and deterministic.

### Phase 2: SQL Repository and Migrations

Deliverables:

- SQL tables, normalized association tables, and indexes.
- Idempotent writes for requirements, availability, assessments, disagreements, lineage, processing runs, and checkpoints.
- Point-in-time query methods.

Acceptance criteria:

- Complete sufficiency lineage reconstructs without JSON/list scans.
- Repeated writes of the same version are idempotent.
- Later assessments preserve prior historical state.
- Existing repositories and runtime behavior remain backward compatible.

### Phase 3: Availability Evaluation

Deliverables:

- Requirement selection by engine, purpose, field, candidate, and checkpoint.
- Availability evaluator for direct, proxy, stale, partial, and unavailable states.
- Integration with Candidate Registry, Identity/Trust, Evidence Intelligence, Competitive Intelligence, and existing provider status as read-only inputs.

Acceptance criteria:

- Missing data returns explicit unavailable state.
- Stale data returns stale state and does not masquerade as live.
- Partial data preserves which requirements are missing.
- Explicit candidate selection cannot bypass Identity/Trust gating.

### Phase 4: Cross-Source Validation and Disagreement Handling

Deliverables:

- Source compatibility rules.
- Cross-source validation records.
- Disagreement records with lineage and authority context.
- Distinction between conflict, disagreement, unavailable provider, and stale source.

Acceptance criteria:

- Different scopes, chains, products, periods, or units do not create false disagreements.
- Disagreements are data-quality states, not negative project evidence.
- Source authority and lifecycle state are replay-safe.

### Phase 5: Sufficiency Assessment and Degraded Mode

Deliverables:

- Sufficiency aggregation service.
- Degraded-mode policy engine.
- Limitation text generation from deterministic rules.
- Report-field gating or annotation metadata.

Acceptance criteria:

- Sufficiency state remains separate from analytical conclusions.
- Degraded mode never substitutes proxy values for direct observations.
- Blocked outputs explain the exact missing requirements.
- Current and historical assessments use the requested replay context.

### Phase 6: Reporting, CLI, Automation, and Documentation

Deliverables:

- `hunter sufficiency` CLI namespace.
- Coverage, requirement, assessment, missing-evidence, disagreement, and report outputs.
- Idempotent automation install/status.
- Documentation update for sufficiency states, proxy semantics, degraded mode, replay, and limitations.

Acceptance criteria:

- Reports expose material missing evidence and analysis limitations.
- Automation jobs are valid existing Scheduler job definitions.
- No analytical logic is placed in Scheduler.
- No scoring, ranking, committee, Opportunity Timing, Market Validation, valuation, or canonical runtime behavior changes.

## Risks

- False confidence from proxy signals: mitigated by strict direct/proxy labeling and policy enforcement.
- Treating missing data as negative evidence: mitigated by explicit unavailable and degraded states.
- Coverage inflation: mitigated by separate direct, proxy, stale, partial, unavailable, and historical coverage metrics.
- Lookahead leakage: mitigated by effective and recorded time constraints across requirements, evidence, authority, availability, and assessments.
- Report complexity: mitigated by concise limitations and material missing-evidence summaries.
- Scale pressure: mitigated by indexed requirement/candidate selection, checkpoints, batching, and incremental refresh.
