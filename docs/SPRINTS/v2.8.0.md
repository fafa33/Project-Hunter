## Read First

The sprint followed the Hunter architecture manifest, architecture specification, roadmap, implementation guide, Discovery Layer documentation, and `v2.8.0` sprint instructions.

## Mission

Expand Hunter's discovery capability from a limited provider set into a broader market acquisition platform while preserving the existing Discovery Layer architecture.

## Business Objective

Increase practical market visibility so Hunter can discover and triage more investable assets, protocols, ecosystems, and DEX-market candidates before Identity Resolution begins.

## Scope

- GeckoTerminal discovery adapter.
- DexScreener discovery adapter.
- Chain-plus-contract identifiers for DEX-market candidates.
- Cross-provider candidate merging by deterministic contract identity when available.
- Provider environment-variable overrides.
- Expanded discovery reporting for provider, ecosystem, chain, category, overlap, uniqueness, new candidates, and coverage growth.
- Discovery source-health automation job.
- Documentation updates for public sources, endpoint overrides, reporting, and DEX-market identity policy.
- Deterministic tests for new adapters and contract-identity merge behavior.

## Out of Scope

- Identity Resolution.
- Intrinsic Value.
- Competition Intelligence.
- Network Effects.
- Machine Learning.
- Portfolio Engine.
- REST API.
- Dashboard redesign.
- Canonical runtime, scoring, weighting, committee, timing, replay, historical validation, or calibration changes.

## Engineering Principles

- Reuse the existing provider adapter framework.
- Add only providers that materially improve market coverage or evidence quality.
- Preserve deterministic behavior and source provenance.
- Do not treat DEX-market listings as proof of full project identity.
- Do not force provider disagreements into resolved identities.

## Performance

- Maintain indexed registry lookups.
- Use chain-plus-contract identifiers for scalable DEX-market overlap detection.
- Avoid full pairwise candidate comparison.
- Keep provider sync bounded by configured limits.

## Automation

- Add discovery source-health automation job.
- Preserve idempotent automation installation.
- Keep Scheduler operational-only.
- Keep discovery execution in the existing Discovery Layer.
