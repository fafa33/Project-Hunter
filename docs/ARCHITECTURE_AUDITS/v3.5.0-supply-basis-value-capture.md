# Independent Architecture Audit — v3.5.0 Supply Basis and Value-Capture Evidence Foundation

## Audit Status

- Release audited: `v3.5.0`
- Canonical target: `main` after PR #41
- Audit result: **BLOCKED**
- Cleanup result: temporary mypy diagnostic and autofix workflows removed on the audit branch

The implementation establishes a useful additive foundation, but it does not yet satisfy the full authority and replay guarantees required by the approved sprint and ADR 0021. The next valuation sprint must not start until the blockers below are fixed and re-audited.

## Verified Foundations

### V1 — Immutable typed record families

Verified.

The implementation provides immutable dataclasses for:

- `FundamentalEvidenceRecord`
- `SupplyBasisSnapshot`
- `ValueCaptureRuleSnapshot`
- `EconomicClaimIdentity`

The models preserve explicit entity, economic-claim, asset, token, representation, chain, contract, effective-time, recorded-time, known-time, quality, conflict, correction, and evidence-link fields.

### V2 — Dedicated SQL-authoritative persistence

Verified with limitations described below.

The implementation creates dedicated tables and migration metadata for evidence, supply-basis snapshots, and value-capture rules. Repository writes are transactional and insert-or-identical behavior is present for exact `record_id` retries.

### V3 — Category separation

Verified.

Supply categories and value-capture rule types are explicitly typed and are not automatically interchangeable. Total supply does not automatically satisfy circulating supply, and wrapped/bridged representations remain separate through canonical identity.

### V4 — Strict cutoff filtering

Verified with lineage and evidence-time limitations described below.

Repository selectors enforce:

- `effective_at <= effective_as_of`
- `recorded_at <= known_by`
- `known_at <= known_by`
- accepted quality only
- no open or contested conflict

### V5 — Explicit exclusion boundary

Verified.

The implementation does not activate fair value, comparative valuation, mispricing, asymmetry, Opportunity scoring, ranking, Portfolio Intelligence, Dashboard calculations, or Market Validation scalar inputs.

## Blocking Findings

### B1 — Registry fingerprint is not acquisition provenance

**Severity: Critical**

`SupplyAndValueCaptureService` accepts caller-constructed records together with a public deterministic registry fingerprint, endpoint string, source ID, and parser version. Any caller that can read the registry can reproduce the fingerprint and submit a forged “official” or “on-chain” record.

The service does not require a typed provider result produced by an approved adapter, a registry-issued acquisition context, a request/response evidence envelope, or a verifiable raw acquisition reference.

Consequences:

- forged successful evidence can reach authoritative persistence;
- forged unavailable or unsupported evidence can also be represented as authoritative source output;
- host, endpoint, parser, and capability validation prove only that submitted strings match configuration, not that acquisition occurred;
- the sprint requirement for forged-result rejection is not satisfied.

Required fix:

- introduce typed provider/acquisition result models;
- bind every result to a registry-approved acquisition request/context;
- preserve request identity, response hash, acquisition timestamp, provider identity, parser identity, and registry source fingerprint inside the result;
- allow the service to accept only results created through the approved provider boundary;
- reject caller-authored authoritative records before normalization or persistence.

### B2 — Evidence-time causality is not enforced

**Severity: Critical**

`_require_evidence()` checks only evidence existence, canonical identity, accepted quality, and conflict state. It does not ensure that cited evidence was known by the snapshot's `known_at` or recorded by the snapshot's `recorded_at`.

A supply or value-capture snapshot can therefore cite evidence learned after the snapshot's claimed knowledge cutoff.

Consequences:

- strict-known replay can select a snapshot whose supporting evidence was not actually known at the replay cutoff;
- post-cutoff evidence leakage is possible even though the snapshot's own timestamps pass the repository selector;
- ADR 0020 strict-known semantics are not satisfied end-to-end.

Required fix:

For every cited evidence record, enforce at service authorization time:

- `evidence.effective_at <= snapshot.effective_at` where the evidence semantics require it;
- `evidence.recorded_at <= snapshot.recorded_at`;
- `evidence.known_at <= snapshot.known_at`;
- evidence source/parser/capability compatibility with the derived snapshot;
- evidence lineage valid at the snapshot's cutoff.

Replay selection must also prove that the selected snapshot's complete evidence set was eligible at the requested cutoff.

### B3 — Source and parser linkage between evidence and derived snapshots is incomplete

**Severity: High**

The service permits an accepted evidence record from one source/parser to support a supply or value-capture snapshot submitted under another source/parser, provided canonical identity matches.

No explicit derivation policy defines when cross-source evidence is permitted, what transformation occurred, or which source owns the resulting conclusion.

Consequences:

- evidence authority can be silently relabeled;
- parser provenance can be lost across derivation;
- an aggregator or lower-tier source can support a record presented as an official-source conclusion;
- source authority tiers are not preserved end-to-end.

Required fix:

- add explicit derivation policy and source-compatibility rules;
- persist evidence-source IDs, parser versions, authority tiers, and derivation method on snapshots;
- require same-source derivation by default;
- allow cross-source derivation only through an explicit service-owned registered methodology.

### B4 — Correction authorization is incomplete

**Severity: High**

The repository verifies only that the predecessor exists and shares the same `logical_id`. The service does not require:

- a non-empty correction reason when `supersedes_record_id` is present;
- correction timestamps later than the predecessor's recorded/known timestamps;
- source or authority continuity;
- evidence supporting the correction;
- prevention of multiple competing successors for the same predecessor;
- explicit conflict state when divergent successor branches exist.

Consequences:

- corrections can be backdated;
- multiple incompatible successors can coexist without deterministic lineage resolution;
- correction authority is not fully service-owned;
- current replay ordering can choose a record without proving a single valid lineage.

Required fix:

- authorize corrections in the service after loading the predecessor;
- enforce monotonic recorded/known times;
- require a correction reason and correction evidence;
- enforce source/authority continuity or an explicit transfer policy;
- prevent multiple active successors or represent them as an explicit conflict;
- make replay select by valid lineage, not timestamp ordering alone.

### B5 — Repository authority token is reachable from service consumers

**Severity: Medium**

The service accesses `repository._authority`, and the repository instance is publicly exposed as `service.repository`. Python privacy is conventional only; external code can obtain the same authority object and construct a valid `ValueCaptureWritePlan`.

Consequences:

- direct authoritative mutation is not technically prevented;
- the repository authorization boundary is weaker than the implementation report claims.

Required fix:

- make write-plan creation repository-internal or service-factory-only;
- expose a narrow repository protocol to the service;
- avoid publishing the authority token through public attributes;
- add tests proving consumers cannot construct an accepted write plan.

### B6 — Model-level temporal and correction invariants are incomplete

**Severity: Medium**

The models normalize timezone awareness but do not enforce basic temporal ordering such as:

- `effective_at <= recorded_at` where required;
- `known_at <= recorded_at` or the repository's selected canonical policy;
- correction reason required only with supersession and prohibited otherwise;
- supersession required for correction reason;
- schema version equality with the supported runtime schema.

Required fix:

Define and enforce the canonical temporal policy in the service and model constructors, then add deterministic tests for invalid orderings and malformed correction states.

## Test Coverage Gaps

The focused tests do not currently prove:

- forged provider-result rejection through a typed acquisition boundary;
- evidence known-time eligibility relative to a derived snapshot;
- source/parser/authority compatibility between evidence and snapshot;
- backdated correction rejection;
- duplicate successor conflict handling;
- inability of consumers to obtain repository write authority;
- strict-known selection over branching correction lineage;
- explicit unavailable acquisition outcomes with preserved operational evidence;
- exact ingestion from v3.4.0 observed-market-fact record IDs;
- atomic rollback across a mixed evidence/supply/rule write plan under a forced mid-transaction failure.

## Conclusion

The implementation is a valid structural foundation, but **v3.5.0 is not architecturally closed**.

The release must remain blocked from the next Valuation Methodology sprint until B1 through B4 are fixed. B5 and B6 should be fixed in the same remediation because they directly protect the service-owned authority boundary and deterministic replay semantics.

## Required Next Action

Create a dedicated remediation sprint/PR limited to:

1. typed acquisition/provider result authority;
2. end-to-end evidence-time causality;
3. explicit source/parser derivation policy;
4. correction-lineage authorization and deterministic replay;
5. sealed repository write authorization;
6. missing deterministic tests;
7. removal of all temporary diagnostic/autofix workflows.

Do not implement fair value or any valuation-family output in the remediation.
