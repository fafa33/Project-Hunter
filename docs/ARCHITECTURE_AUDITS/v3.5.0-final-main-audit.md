# v3.5.0 Final Main Audit

Status: **BLOCKED**

Audited target: `main` after merge commit `8385c288c2a222908d01f52628e3494abb943574`.

Scope:

- persistent provider signing identity;
- acquisition receipt integrity and historical verification;
- repository write authority;
- service, registry, provenance, correction, and strict-known replay boundaries;
- exclusion of valuation, mispricing, asymmetry, Opportunity, Portfolio, Dashboard, and Market Validation scalar activation.

## Verified controls

- Provider signing identity is now supplied explicitly through `signing_key_id` and a persistent runtime key instead of being regenerated on each process start.
- Acquisition receipts persist the signing-key identity and signature.
- Provider verification recomputes the payload hash and receipt hash before accepting an acquisition result.
- The previous public `open_authoritative_value_capture_store()` writer closure has been removed.
- Repositories without verification keys reject authoritative writes.
- Receipt and canonical record persistence remain atomic.
- Evidence causal bounds, exact source/parser matching, correction transition policy, single-successor lineage, and strict-known correction replay remain enforced.

## Remaining blockers

### B1 — Repository verification does not recompute the receipt hash

`ValueCaptureVerificationKeyRegistry.verify_receipt()` verifies only that the supplied `receipt.signature` matches the supplied `receipt.receipt_hash`. It does not independently reconstruct the canonical receipt hash from acquisition ID, kind, capability, source, endpoint, parser, registry fingerprint, signing-key identity, acquisition time, identity, and raw payload hash.

`SupplyAndValueCaptureRepository._commit_authoritative()` relies on that verification directly. Therefore repository-side verification is not independently hash-bound to all receipt fields. A receipt whose fields and `receipt_hash` diverge is not rejected at the repository authority boundary when the supplied hash has a valid signature.

Required closure:

- verification-key authority must recompute the canonical receipt hash from the receipt fields;
- verification must reject any mismatch before signature acceptance;
- the canonical hashing implementation must not require a caller-supplied mutable source object;
- add deterministic tests that alter each security-relevant receipt field while retaining the original signed hash and confirm repository rejection.

### B2 — Authoritative repository mutation remains directly reachable

`SupplyAndValueCaptureRepository._commit_authoritative()` is an instance method accessible to any caller holding a repository configured with verification keys. It verifies receipt signature and record binding but does not enforce registry endpoint/capability/source policy, correction-source transition policy, evidence causal authority, or service-owned record normalization.

Consequently a caller can bypass `SupplyAndValueCaptureService` and submit a signed receipt plus a caller-constructed canonical record directly to the repository. The leading underscore is a naming convention, not an authority boundary.

Required closure:

- move authoritative mutation behind a non-exported service-owned capability that cannot be obtained from the public repository constructor;
- keep the public repository read-only;
- ensure the write path performs receipt verification and atomic persistence but is reachable only after service validation;
- add a deterministic test showing that callers with the public repository and verification-key registry cannot invoke authoritative persistence directly.

## Decision

v3.5.0 is not yet approved for formal closure. Valuation Methodology Foundation must not begin until B1 and B2 are remediated, all repository quality gates are green on one stable head, temporary workflows are absent, and a subsequent independent audit returns **APPROVED**.
