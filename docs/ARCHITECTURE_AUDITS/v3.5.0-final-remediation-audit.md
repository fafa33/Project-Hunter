# v3.5.0 Final Remediation Audit

Status: **BLOCKED**

Audited target: `main` after merge of PR #44 (`e59ac6bf10325379fb4cebfdd8faef3f32cc0ff8`).

## Verified improvements

- Authoritative records are now constructed by `SupplyAndValueCaptureService` from typed acquisition results.
- Registry source, endpoint, parser, capability, canonical identity, and representation checks are present.
- `effective_at <= recorded_at <= known_at` is enforced by the models.
- Supporting evidence must not be recorded or known after a derived snapshot.
- Evidence source/parser and identity must match the derived snapshot.
- Correction lineage rejects multiple successors for one predecessor.
- Correction chronology and strict-known correction selection are implemented.
- Focused tests cover forged ordinary construction, future-known evidence, chronology, provenance mismatch, branching correction, and replay cutoffs.

## Remaining blockers

### B1 — Provider acquisition sealing is forgeable from production Python

`src/hunter/value_capture/providers.py` uses module-level `_PROVIDER_SEAL`. Python callers can import that object and call `ValueCaptureAcquisitionResult(..., seal=_PROVIDER_SEAL)`. The test only verifies rejection of an unrelated `object()` and therefore does not prove an external caller cannot forge a sealed result.

Required closure:

- callers must not be able to obtain or replay the authority capability used to create an accepted acquisition result;
- provider output validation must bind to a service-owned acquisition receipt or persisted acquisition authority record, not only an importable in-process sentinel;
- add a deterministic adversarial test that imports all reachable module symbols and still cannot manufacture an accepted acquisition result.

### B2 — Repository write authority remains reachable

`src/hunter/value_capture/service.py` defines module-level `_SERVICE_WRITE_TOKEN`, while `src/hunter/value_capture/repository.py::_service_commit` imports and compares that token. A caller can import the same token and `_service_commit`, or directly use the exposed repository object's `_connect()` method to mutate authoritative tables.

The current test only checks that `apply`, `write`, and `_authority` attributes are absent. It does not prove service-only writes.

Required closure:

- remove any importable reusable write token;
- prevent application callers from obtaining a writable SQL connection through the repository API;
- move mutation behind a construction-time capability that is not exported or reconstructable, or behind a separate persistence owner unavailable to normal callers;
- add adversarial tests for importing internal symbols, invoking `_service_commit`, and direct SQL mutation attempts.

### B3 — Acquisition provenance is not persisted as an independently authoritative receipt

Records persist `acquisition_id`, endpoint-derived authorization, parser, source, and raw payload hash, but no independent acquisition receipt is stored and validated. A forged in-process result that reaches the service can therefore supply its own acquisition ID and payload while appearing registry-consistent.

Required closure:

- persist immutable acquisition receipts before evidence/snapshot construction;
- receipt must include source registry fingerprint, endpoint, parser, capability, acquired time, canonical identity, payload hash, provider implementation/version, and outcome state;
- evidence, supply, and rule records must reference an existing receipt by ID and exact hash-bound fields;
- strict-known replay must require the receipt to have been known by the replay cutoff.

### B4 — Correction authority is structural but not source-authoritative

The repository validates predecessor existence, logical identity, chronology, and single-successor lineage. It does not prove that a correction is supported by a new authoritative acquisition receipt or that the correcting source is permitted to supersede the predecessor's authority class.

Required closure:

- every correction must reference a distinct authoritative acquisition receipt;
- correction source/authority transition rules must be explicit and registry-governed;
- prohibit a lower-authority or unrelated source from superseding a higher-authority record unless an explicit policy allows it;
- test predecessor-source mismatch, reused acquisition receipt, and unauthorized authority transitions.

## Decision

PR #44 materially improves the v3.5.0 implementation, but the original requirement that authoritative provider results and repository writes be unforgeable outside the service boundary is not yet satisfied.

Do not begin Valuation Methodology implementation until B1-B4 are closed and independently re-audited.

## Scope boundary

This audit and cleanup change no production models, services, repositories, schemas, valuation outputs, Opportunity logic, ranking, Portfolio Intelligence, Dashboard calculations, or Market Validation weights.
