# v3.5.0 Post-Merge Final Architecture Audit

Status: **BLOCKED**

Audited target: `main` after merge of PR #46 (`a7a7bdc95ad3a2eb1758ef9e8ef2f9aaa4f08370`).

## Scope

This audit reviewed the production authority boundary for value-capture acquisition, receipt verification, receipt persistence, canonical record persistence, correction governance, and strict-known replay. It does not evaluate valuation methodology, comparative valuation, mispricing, asymmetry, Opportunity ranking, Portfolio Intelligence, Dashboard calculations, or Market Validation weighting.

## Verified improvements

- Provider results are signed with per-provider runtime keys.
- Service ingestion verifies provider origin, payload hash, receipt hash, registry fingerprint, endpoint, parser, capability, source, and acquisition kind.
- Acquisition receipts are persisted in a dedicated SQL table.
- Receipt and derived record persistence occur in one transaction.
- Records are bound to receipts by acquisition ID, payload hash, source, parser, identity, and chronology.
- Correction transitions are checked against registry authority-tier policy.
- Branching correction lineage and strict-known replay behavior remain enforced.
- The merged PR passed the repository quality gates.

## Blocking findings

### B1 — Authoritative writer remains publicly obtainable

`open_authoritative_value_capture_store()` is a public module-level function. Any caller can import it and receive the closure-scoped `AuthorityWriter`. The writer accepts caller-constructed `AcquisitionReceipt` and canonical record objects and does not require a verified provider result or service authorization.

Evidence:

- `src/hunter/value_capture/repository.py`: public `open_authoritative_value_capture_store()` returns `(repository, commit)`.
- The returned `commit` performs binding checks and SQL insertion but does not verify provider signature or registry authorization.

Impact: the service is not the exclusive production write authority. A caller can bypass provider signature verification and persist a self-consistent forged receipt/record pair.

Required closure:

- Remove the public writer factory from the importable repository surface.
- Establish composition-owned write authority that cannot be obtained by arbitrary application callers.
- Add a deterministic test proving direct writer acquisition and forged receipt persistence are impossible.

### B2 — Persisted receipt is not independently authenticity-verifiable

The SQL receipt table persists receipt fields and `receipt_hash`, but not the provider signature or a durable verification key identity. Repository persistence validates only receipt-to-record consistency. After process restart, the persisted receipt cannot independently prove that the originating registered provider signed it.

Impact: the receipt is an integrity record, not an independently authoritative acquisition receipt. A forged but internally consistent receipt can satisfy repository binding checks if the public writer is reached.

Required closure:

- Persist signature material and a stable provider verification-key identifier, or use another durable verification mechanism.
- Verify the signed receipt at the authoritative persistence boundary before insertion.
- Define key rotation and replay semantics so historical receipts remain verifiable.

### B3 — Runtime-generated provider keys are not stable authority identities

`RegisteredValueCaptureProvider` generates a random HMAC key during object construction. Reconstructing the provider creates a new key. Historical signatures therefore cannot be revalidated after restart, and two instances for the same registry source represent different signing authorities without an explicit key identity.

Impact: provider authenticity is process-local rather than durable and replayable.

Required closure:

- Bind each registered provider to a stable, configuration-governed verification identity.
- Do not persist raw secret keys in repository records.
- Support controlled key rotation with immutable key IDs and validity intervals.

## Non-blocking observations

- `SupplyAndValueCaptureService` accepts an existing repository object but reopens the store from `repository.path`; this is surprising composition behavior and should be simplified during blocker remediation.
- Receipt uniqueness by `acquisition_id` is correctly enforced across record families.

## Decision

v3.5.0 is **not architecture-closed**. PR #46 materially improves the implementation, but the production authority boundary remains bypassable and receipt authenticity is not durable across process restarts.

Do not begin Valuation Methodology implementation until B1–B3 are closed and independently re-audited. The next change must be a narrow authority-composition and durable receipt-verification remediation, with no valuation or scoring activation.
