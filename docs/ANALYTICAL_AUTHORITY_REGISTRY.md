# Project Hunter Analytical Authority Registry

**Status:** Canonical documentation-first authority registry

**Authority:** ADR 0016

**Scope:** Current analytical and analytical-looking outputs

**Implementation effect:** None; this document defines ownership and classification only

## Purpose

This registry is the canonical documentation map for who may define each analytical or analytical-looking output in Project Hunter. It implements Phase 1.2 of `docs/VALIDATED_IMPLEMENTATION_PLAN.md` under `docs/ADR/0016-runtime-analytical-authority.md`.

The existence of a model, engine, test, CLI command, automation job, record class, repository, JSONL row, SQLite table, Dashboard field, or desktop view never proves production authority. Production authority exists only where an accepted ADR assigns it.

## Registry Rules

- Each output has one classification and one semantic owner. Where authority or a unified semantic contract is absent, the owner is explicitly `none` and no component may claim that authority.
- `production` means a current authoritative analytical output. ADR 0011–0015 findings are production descriptive outputs but are not canonical scores or investment conclusions.
- `experimental` means research infrastructure that cannot be presented to production consumers as authoritative.
- `operational-only` means lifecycle, status, audit, or read-projection data with no analytical decision authority.
- `unimplemented authority` means no current service, persistence contract, lifecycle, or consumer may claim the output.
- Repositories store service-authorized records. They never own meaning, clocks, replay selection, scoring, ranking, conflict resolution, or lifecycle decisions.
- Dashboard API and the desktop OperationalConsole are read-only consumers. Missing or unavailable authority remains unavailable; experimental output remains labeled experimental.
- Effective time is when an analytical statement applies. Recorded time is when Hunter persisted it. Known time is the latest time by which Hunter could have known the contributing evidence. A row that lacks an explicit known-time contract cannot claim strict-known replay safety.
- Promotion or migration follows ADR 0016. This registry cannot promote an output and cannot create a parallel owner.

## Cross-Cutting Input Authority

Every future consumer of a registered output must also preserve these upstream owners:

- Candidate identity and lifecycle: SQL-backed Candidate Registry under ADR 0003.
- Trust, conflicts, freshness and unavailable state: upstream service-owned trust/evidence flows under ADR 0004.
- Economic entity and representation context: ADR 0005; project, protocol, token, network, contract, wallet and listing are not interchangeable.
- Evidence provenance and replay cutoff: ADR 0002.
- Persistence authorization: Provider → Service → Repository → Persistence under ADR 0009.
- Descriptive Intelligence Engine boundaries: ADR 0010–0015; findings remain independent and non-scoring.

No knowledge graph is authorized by this registry. ADR 0006 keeps existing Candidate Registry and evidence repositories authoritative until identity and trust foundations are production-stable and a future ADR approves graph-runtime work.

## Promotion And Change Control

Classification or ownership changes require an accepted ADR satisfying ADR 0016's promotion and migration rules. A change must define semantic purpose, input authority, provenance, effective/recorded/known time, replay, lifecycle, historical validation, compatibility/cutover, persistence authorization and permitted API/Dashboard exposure.

This document may be updated to reflect an accepted decision or verified implementation state. It cannot itself promote an output, select a database, authorize persistence, or alter runtime behavior.
